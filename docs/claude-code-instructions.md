## プロジェクトの概要

`https://your-doctor.jp/media/` のWordPressサイトに、AGAメディア立ち上げのための技術基盤を実装します。

具体的な実装対象は以下3点：

1. 監修医師のカスタム投稿タイプ（CPT）と関連機能のプラグイン化
2. 記事のターゲットKW記録機能
3. 構造化データ（JSON-LD）の自動出力（AIOSEOと連携）

## 必須読み込みドキュメント

作業着手前に、以下のドキュメントを必ず順に熟読してください：

1. `README.md` - プロジェクト全体像
2. `docs/design.md` - 詳細設計書 v1.0（最重要）
3. `docs/setup-guide.md` - 環境セットアップ手順（参考）

特に `docs/design.md` は本プロジェクトの仕様の唯一の正です。実装で迷ったらこの設計書に従ってください。設計書と異なる判断が必要と思われる場合は、独断で進めず必ず私に確認してください。

## 開発環境の制約

このプロジェクトには以下の制約があります：

- **ローカルWP環境はありません**：動作確認は本番サイト（`/media/`）に直接デプロイして行います
- **本番反映は手動FTP（初期段階）**：Claude Codeが直接サーバーにファイルを送ることはできません。コードは私が手動で本番にアップロードします
- **サーバーへのSSHアクセスはありません**：開発はすべてローカルファイルシステム上で完結します
- **PHP実行環境**：ローカルにPHPがインストールされていれば構文チェックが可能（`php -l ファイル名`）

## 既存リソース

- `theme/swell_child_custmon/` には、既に本番に適用されている子テーマファイルが配置されています（style.css / functions.php / header.php / footer.php の4本）。新規プラグインの実装と、必要に応じてこの子テーマの修正を行います。
- 親サイト `https://your-doctor.jp/` は別の完全カスタムテーマで動作しています。子テーマはこの親サイトのデザインを再現する形で構築されています。

## コーディング規約

### PHP
- WordPress Coding Standards に準拠
- インデント：タブ（4幅）
- ファイル末尾に空行
- `<?php` のみ（閉じタグ `?>` は省略）
- 関数・クラスにはdocblockコメント
- 直接アクセス防止：すべてのPHPファイル冒頭に `if ( ! defined( 'ABSPATH' ) ) exit;`
- 関数・クラスにプレフィックス `yd_`（衝突回避）

### 命名規則
- 関数：`yd_get_post_supervisors()`（snake_case + プレフィックス）
- クラス：`YD_CPT_Doctor`（大文字スネーク + プレフィックス）
- 定数：`YD_SUPERVISOR_VERSION`
- メタキー：`yd_target_keyword`（プレフィックス付き）
- CPT名：`yd_doctor`
- フックタグ：`yd_supervisor_*`

### セキュリティ
- ユーザー入力は必ずサニタイズ（`sanitize_text_field`, `esc_url_raw`等）
- 出力時はエスケープ（`esc_html`, `esc_attr`, `esc_url`等）
- nonce検証
- 権限チェック（`current_user_can`）

### i18n
- 文字列は `__('テキスト', 'yd-supervisor')` で翻訳可能にする
- テキストドメインは `yd-supervisor`

## 注意事項

### AIOSEO連携について
構造化データの実装では、AIOSEOの公式フィルターを使います。フィルター名・引数の正確な仕様は、実装時にAIOSEOの公式ドキュメントまたはソースコードで確認してください。私の知識は古い可能性があるため、推測で実装せず、最新版を確認してから着手してください。

### ACF Pro前提の機能利用
ACF Pro固有の機能（repeater, post_object, wysiwyg等）を使う場合、Free版では動作しないことを意識してください。本プロジェクトではPro版前提で問題ありません。

### 本番影響への配慮
プラグインを有効化した瞬間に問題が起きないよう、Phase 1では「管理画面のみへの影響」を確認してから次に進みます。フロントエンドへの変更は最終フェーズに集約します。

## 動作確認できないことへの対処

ローカル環境がないため、各フェーズ実装後にDaikiさんが本番にアップロードして動作確認します。Claude Codeとしては：

- PHPの構文エラーは絶対に出さないこと（`php -l` で必ず確認）
- 本番環境での副作用（既存記事の崩れ、エラー画面）を起こす可能性があるコードは特に注意
- 実装と並行して `docs/test-checklist.md` のような検証手順書を作成し、私が動作確認できるようにする

## 補足

### このプロンプトの設計意図

- **設計書を「唯一の正」として明示**：Claude Codeが独自判断で逸脱しないよう
- **動作確認の制約を事前共有**：ローカル環境なしの前提を理解させる
- **段階的な進行を強制**：Phase毎の確認サイクルを組み込む
- **コーディング規約の明示**：プロジェクト全体の一貫性を担保
- **読み込みの強制**：着手前に必ず3ファイルを読ませる

### 進行が始まった後の運用

Claude Codeから報告が来たら、以下の流れで進めます：

1. 報告内容を確認
2. 質問があれば回答
3. 設計書の未確定論点で判断が必要なら都度判断
4. Phase完了報告 → 本番にFTPアップロード → 動作確認 → 次Phase着手承認

### 注意：このプロンプトを送信する前に

`docs/design.md` と `theme/swell_child_custmon/` の4ファイルが正しく配置されていることを確認してください。これらが揃っていない状態で開始すると、Claude Codeが情報不足のまま動作してしまいます。
